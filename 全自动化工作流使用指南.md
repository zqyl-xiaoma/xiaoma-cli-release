# 🚀 全自动化工作流使用指南

本指南详细说明如何使用XiaoMa-CLI的自动化用户故事开发工作流,实现 SM→PO→Dev→QA 的完全自动化开发流程。

---

## 📋 一、前置条件准备 (必须完成)

在启动全自动化工作流之前,必须确保以下前置条件已满足:

### 1️⃣ **Epic分解**

```bash
# 检查Epic文件是否存在
ls -la docs/epics/

# 如果不存在,使用PM或PO智能体创建Epic
/pm
*create-epic
```

**要求**: 必须存在Epic分解文件 (\*.md)
**文件位置**: `docs/epics/`

### 2️⃣ **数据库设计**

```bash
# 检查数据库设计文档
ls -la docs/database/database-design.md

# 如果不存在,使用database-architect完成设计
/database-architect
*design-database
```

**要求**: 必须存在完整的数据库设计文档
**文件位置**: `docs/database/database-design.md`

### 3️⃣ **生成基础代码**

```bash
# 检查生成的Java实体类
ls -la src/main/java/

# 如果不存在,使用database-architect生成
/database-architect
*generate-entities
```

**要求**: 必须已生成基础代码(Entity、Mapper、Service)
**目录位置**: `src/main/java/`

### 4️⃣ **Stories目录**

```bash
# 工作流会自动创建此目录,无需手动操作
docs/stories/
```

**说明**: 工作流会在首次运行时自动创建此目录

## 🎯 二、启动全自动化工作流

### 方式1: 完全自动化模式 (✨ 推荐)

```bash
# 1. 切换到自动化编排器智能体
/automation-orchestrator

# 2. 启动自动化用户故事开发流程
*start-auto-development
```

#### 这个命令会自动执行:

- ✅ 检查所有前置条件(Epic、数据库设计、基础代码)
- ✅ 初始化流程状态和进度跟踪
- ✅ 启动第一个用户故事开发循环
- ✅ 监控整个流程执行
- ✅ 自动完成 SM→PO→Dev→QA 循环
- ✅ 循环处理所有用户故事直到完成

---

### 方式2: 单个故事循环模式

```bash
# 切换到自动化编排器
/automation-orchestrator

# 执行单个用户故事的完整开发循环
*execute-story-cycle
```

#### 这个命令会执行单个故事的完整开发流程:

1. SM智能体创建用户故事
2. PO智能体验证故事质量
3. Dev智能体开发和自测
4. QA智能体测试验证

## ⚠️ 五、重要注意事项

### 🔴 核心测试原则

**QA测试必须使用真实数据,严禁mock数据**:

| ❌ 错误做法                         | ✅ 正确做法                                     |
| ----------------------------------- | ----------------------------------------------- |
| 使用mock数据返回假的查询结果        | 调用查询API → 验证返回数据与数据库实际一致      |
| mock数据库返回成功,实际没有插入数据 | 调用创建API → 查询数据库验证记录真实插入        |
| mock返回成功,实际数据库没有变化     | 调用更新API → 查询数据库验证记录真实更新        |
| mock返回成功,实际数据库记录仍存在   | 调用删除API → 查询数据库验证记录真实删除/软删除 |

**测试要求**:

- ✅ 查询接口必须**真实从数据库**查询
- ✅ 操作接口必须**真实操作到数据库**
- ✅ 验证**实际的数据库状态变化**
- ✅ 端到端验证整个数据流

### 📝 智能体责任边界

| 智能体                     | 职责                                   | 严格禁止                                                     | 可更新的故事部分                                                       |
| -------------------------- | -------------------------------------- | ------------------------------------------------------------ | ---------------------------------------------------------------------- |
| **SM** (Scrum Master)      | 创建用户故事、提取需求、任务分解       | ❌ 实施代码<br>❌ 修改已批准的故事                           | -                                                                      |
| **PO** (Product Owner)     | 验证故事质量、反幻觉检查、GO/NO-GO决策 | ❌ 创建故事<br>❌ 实现代码<br>❌ 测试                        | -                                                                      |
| **Dev** (Developer)        | 实现代码、编写测试、应用QA修复         | ❌ Story/AC/Dev Notes/Testing<br>❌ QA Results(QA专属)       | ✅ Tasks/Subtasks<br>✅ Dev Agent Record<br>✅ Change Log<br>✅ Status |
| **QA** (Quality Assurance) | 功能测试、代码审查、质量门控决策       | ❌ Story/AC/Tasks/Dev Notes<br>❌ Status/File List(只能推荐) | ✅ QA Results Section Only                                             |

### ✅ 四级质量门控体系

#### 1️⃣ PO质量门 (Story Validation Gate)

**标准**:

- 故事完整性(所有必需章节)
- 反幻觉验证(无虚构内容)
- 实现准备度评分 ≥ 8/10

**决策**: GO / NO-GO

---

#### 2️⃣ Dev质量门 (Development Completion Gate)

**标准**:

- 所有测试通过(单元测试、集成测试)
- 代码覆盖率 ≥ 80%
- Maven编译成功
- 代码质量检查通过(checkstyle/pmd/spotbugs)

**决策**: Ready for Review

---

#### 3️⃣ QA质量门 (QA Approval Gate)

**标准**:

- 功能完整性(所有验收标准满足)
- 真实数据测试通过(无mock数据)
- API契约有效
- 数据完整性保持
- 业务逻辑正确

**决策**: PASS / CONCERNS / FAIL / WAIVED

---

#### 4️⃣ 最终门控 (Story Completion Gate)

**标准**:

- QA Gate = PASS
- 所有验收标准满足
- 无阻塞性问题

**决策**: Done ✅

## 📚 八、常见问题 (FAQ)

### Q1: 如果前置条件不满足会怎样?

**A**: 工作流会在预检阶段停止,并提示您完成缺失的前置工作:

```
❌ 环境验证失败,请完成以下前置工作:
1. 使用PM完成Epic分解
2. 使用database-architect完成数据库设计
3. 使用database-architect生成基础代码
4. 确保所有依赖文件就绪后重新启动工作流
```

### Q2: 工作流执行过程中出现错误怎么办?

**A**: 工作流内置了智能错误处理机制:

- **编译失败**: 自动尝试修复Maven配置、Lombok设置,最多重试2次
- **测试失败**: Dev自动修复,最多5次尝试
- **QA失败**: 自动返回Dev进行确定性修复,循环直到通过

### Q3: 如何知道当前进度?

**A**: 随时可以查看进度:

```bash
/automation-orchestrator
*generate-progress-report
*check-completion-status
```

### Q4: 可以中途暂停吗?

**A**: 可以。每个用户故事的状态会保存在story文件中,下次启动时会从上次中断的地方继续。

### Q5: 为什么强调不能使用mock数据?

**A**: 真实数据测试的重要性:

- ✅ 确保功能在实际环境中可用
- ✅ 发现真实数据库操作中的问题
- ✅ 验证数据完整性和一致性
- ❌ Mock数据可能导致假阳性(测试通过但实际有问题)

## 🔧 九、高级配置

### 自定义质量标准

可以在工作流配置中调整质量标准:

```yaml
# xiaoma-core/workflows/automated-story-development.yaml

quality_gates:
  development_completion:
    criteria:
      code_coverage_target: '>=80%' # 可调整覆盖率目标
      performance_acceptable: true
```

### 调整重试策略

```yaml
error_handling:
  development_failure:
    max_attempts: 5 # 可调整最大尝试次数

  qa_failure:
    impact_analysis: true # 评估对其他故事的影响
```
